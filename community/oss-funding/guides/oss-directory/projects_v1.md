## Column Relationships

- **`project_id` is deterministic** — generated by `@oso_entity_id(project_source, project_namespace, project_name)`. All OSO entity IDs follow this pattern: hash the (source, namespace, name) tuple. The same project registered in two different sources (eg OSS Directory and OP Atlas) will have two different `project_id` values and appear as separate rows.
- **`project_source` controls which registry the project comes from** — the same real-world project may appear under multiple sources. Filter by `project_source = 'OSS_DIRECTORY'` to get OSS Directory projects specifically.
- **`project_name` is unique within a source** but not globally — two different sources can have different slugs for the same real-world project.

## Query Rules

### Filtering Rules

- **Always filter by `project_source`** when working with OSSD data. Use `WHERE project_source = 'OSS_DIRECTORY'` to scope to OSS Directory projects. Without this filter, you will get duplicate entries for projects that appear in multiple registries.
- **Use `display_name` for human-readable searches** — `project_name` is a slug (eg `uniswap`), while `display_name` is the formatted name (eg `Uniswap`).
- **No partition filtering is available** — this is a FULL table. All predicates are post-scan filters.

### Aggregation Rules

- **`COUNT(*)` per `project_source`** to see how many projects each registry contributes.
- **Always include `project_source` in GROUP BY** when counting projects to avoid double-counting across registries.

### Join Rules

- **Join to `oso.artifacts_by_project_v1` on `project_id`** to find all artifacts (repos, packages, contracts) owned by a project.
- **Join to `oso.projects_by_collection_v1` on `project_id`** to find which collections a project belongs to.
- **Cardinality**: one row per (project_source, project_name). Joining to event or metric tables on `project_id` will fan out rows.

### Common Patterns

**Look up a project by name (OSSD only):**
```sql
SELECT
  project_id,
  project_name,
  display_name,
  description
FROM oso.projects_v1
WHERE project_source = 'OSS_DIRECTORY'
  AND project_name = 'uniswap'
```

**Count projects per source registry:**
```sql
SELECT
  project_source,
  COUNT(*) AS project_count
FROM oso.projects_v1
GROUP BY project_source
ORDER BY project_count DESC
```

**Search projects by display name:**
```sql
SELECT
  project_id,
  project_source,
  project_name,
  display_name
FROM oso.projects_v1
WHERE project_source = 'OSS_DIRECTORY'
  AND LOWER(display_name) LIKE '%uniswap%'
```

## Business Logic

This is a thin projection over `int_projects`, which unions project records from multiple upstream registries:

- **OSS Directory** (`stg_ossd__current_projects`) — curated YAML files in the oss-directory repo
- **OP Atlas** (`stg_op_atlas_project`) — Optimism's project registry
- **OpenDevData** (`int_artifacts_by_project_in_opendevdata`) — Electric Capital's developer classification
- **DeFiLlama** (`int_artifacts_by_project_in_defillama`) — DeFi protocol metadata
- **Open Labels Initiative** (`int_artifacts_by_project_in_openlabelsinitiative`) — community-sourced labels

Each source contributes projects independently. The same real-world project (eg Uniswap) may appear under multiple sources with different `project_id` values. There is no cross-source deduplication at this layer — consumers must filter by `project_source` to avoid double-counting.

## Lineage

### Downstream (known consumers)
- `oso.artifacts_by_project_v1` — joins projects to their artifacts
- `oso.projects_by_collection_v1` — joins projects to collections
- `oso.timeseries_metrics_by_project_v0` — metrics computed per project

## Best Practices

- **Always filter by `project_source = 'OSS_DIRECTORY'`** when working with OSSD ecosystem data. Without this filter, you will get projects from OP Atlas, DeFiLlama, and other registries mixed in, leading to duplicates and inflated counts.
- **Use `project_id` for joins**, not `project_name` — names are only unique within a source, and the ID is the canonical join key across all OSO tables.
- **Use `display_name` for presentation** — `project_name` is a machine-friendly slug.
- **This table is metadata only** — it does not contain artifacts, metrics, or events. Join to downstream tables for those.
