## Column Relationships

- **(dependent_artifact_id, package_artifact_id) is the natural key** — one row per dependency relationship. A repo can depend on many packages, and a package can be depended on by many repos.
- **`dependent_artifact_id` maps to `artifact_id` in `repositories_v0` and `artifacts_by_project_v1`** — this is the repo side of the dependency.
- **`package_artifact_id` maps to `artifact_id` in `package_owners_v0`** — this links to who maintains the package.
- **`package_artifact_source` identifies the package ecosystem** — NPM, PyPI, Crates, Go, etc.

## Query Rules

### Filtering Rules

- **Filter by `package_artifact_source`** to scope to a specific package ecosystem (eg `WHERE package_artifact_source = 'NPM'`).
- **Filter by `dependent_artifact_id`** to find all dependencies of a specific repo.
- **Filter by `package_artifact_id`** to find all repos that depend on a specific package.
- **No partition filtering** — FULL table, all predicates are post-scan.

### Aggregation Rules

- **`COUNT(DISTINCT package_artifact_id)`** per dependent repo for dependency count.
- **`COUNT(DISTINCT dependent_artifact_id)`** per package for reverse-dependency count (how widely used a package is).
- **GROUP BY `package_artifact_source`** for breakdown by package ecosystem.

### Join Rules

- **Join to `oso.artifacts_by_project_v1` on `dependent_artifact_id = artifact_id`** to attribute dependencies to projects. Use `WHERE project_source = 'OSS_DIRECTORY' AND artifact_source = 'GITHUB'` on the join.
- **Join to `oso.repositories_v0` on `dependent_artifact_id = artifact_id`** to get repo metadata for the dependent.
- **Join to `oso.package_owners_v0` on `package_artifact_id`** to find who maintains the packages a repo depends on.
- **Cardinality**: many-to-many. Expect significant fan-out when joining repos to their dependencies.

### Common Patterns

**List all dependencies of a specific repo:**
```sql
SELECT
  package_artifact_source,
  package_artifact_namespace,
  package_artifact_name
FROM oso.sboms_v0
WHERE dependent_artifact_name = 'v3-core'
  AND dependent_artifact_namespace = 'uniswap'
ORDER BY package_artifact_source, package_artifact_name
```

**Most depended-on packages across OSSD repos:**
```sql
SELECT
  s.package_artifact_source,
  s.package_artifact_name,
  COUNT(DISTINCT s.dependent_artifact_id) AS dependent_repo_count
FROM oso.sboms_v0 AS s
INNER JOIN oso.artifacts_by_project_v1 AS a
  ON s.dependent_artifact_id = a.artifact_id
WHERE a.project_source = 'OSS_DIRECTORY'
  AND a.artifact_source = 'GITHUB'
GROUP BY s.package_artifact_source, s.package_artifact_name
ORDER BY dependent_repo_count DESC
LIMIT 50
```

**Dependency overlap between two projects:**
```sql
SELECT
  s.package_artifact_source,
  s.package_artifact_name
FROM oso.sboms_v0 AS s
INNER JOIN oso.artifacts_by_project_v1 AS a
  ON s.dependent_artifact_id = a.artifact_id
WHERE a.project_source = 'OSS_DIRECTORY'
  AND a.project_name = 'uniswap'
  AND a.artifact_source = 'GITHUB'
INTERSECT
SELECT
  s.package_artifact_source,
  s.package_artifact_name
FROM oso.sboms_v0 AS s
INNER JOIN oso.artifacts_by_project_v1 AS a
  ON s.dependent_artifact_id = a.artifact_id
WHERE a.project_source = 'OSS_DIRECTORY'
  AND a.project_name = 'aave'
  AND a.artifact_source = 'GITHUB'
```

**Dependency count by package ecosystem for a project:**
```sql
SELECT
  s.package_artifact_source,
  COUNT(DISTINCT s.package_artifact_id) AS package_count
FROM oso.sboms_v0 AS s
INNER JOIN oso.artifacts_by_project_v1 AS a
  ON s.dependent_artifact_id = a.artifact_id
WHERE a.project_source = 'OSS_DIRECTORY'
  AND a.project_name = 'optimism'
  AND a.artifact_source = 'GITHUB'
GROUP BY s.package_artifact_source
ORDER BY package_count DESC
```

## Business Logic

This mart is a `SELECT DISTINCT` over `int_sbom_to_packages`, which resolves SBOM entries to concrete package artifacts. The upstream pipeline works as follows:

1. **SBOM extraction** (`int_sbom_from_ossd`) — parses dependency manifests (package.json, requirements.txt, Cargo.toml, go.mod, etc.) from OSSD-registered repositories via `stg_ossd__current_sbom`.
2. **Package resolution** (`int_sbom__latest_snapshot`) — takes the most recent SBOM snapshot per repo.
3. **Package matching** (`int_sbom_to_packages`) — joins SBOM entries to known packages from deps.dev (`int_packages_from_deps_dev`) using `seed_package_source_mapping` to map between package manager naming conventions and OSO artifact identifiers.

The `DISTINCT` at the mart layer removes any duplicate edges that may arise from multiple SBOM snapshots or resolution paths.

## Lineage

### Full Chain
```
OSSD repo manifests → oso.stg_ossd__current_sbom
                        → oso.int_sbom_from_ossd
                          → oso.int_sbom__latest_snapshot ──┐
deps.dev package data → oso.stg_deps_dev__packages          │
                        → oso.int_packages_from_deps_dev    │
                          → oso.int_packages__current_maintainer_only
                                                            │
oso.seed_package_source_mapping ────────────────────────────┼→ oso.int_sbom_to_packages
                                                               → oso.sboms_v0
```

### Upstream (direct)
- `oso.int_sbom_to_packages` — joins SBOM entries to resolved package artifacts

### Downstream (known consumers)
- Dependency analysis and supply chain mapping
- Package influence scoring
- Ecosystem dependency graphs

## Best Practices

- **Always join to `artifacts_by_project_v1`** to scope dependencies to specific projects — `sboms_v0` alone only tells you which repos depend on which packages, without project attribution.
- **Use `dependent_artifact_id` for repo-side joins** and `package_artifact_id` for package-side joins.
- **Filter by `package_artifact_source`** when analyzing a specific ecosystem (eg NPM for JavaScript dependencies).
- **Expect high fan-out** — a typical repo has dozens to hundreds of dependencies. Aggregations should use `COUNT(DISTINCT ...)` to avoid over-counting.
- **Coverage is OSSD-scoped** — only repositories registered in OSS Directory have their SBOMs parsed. Repos from other registries are not included.
