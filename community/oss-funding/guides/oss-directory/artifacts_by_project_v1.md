# artifacts_by_project_v1

> Maps projects to their artifacts (repos, packages, contracts, addresses). The primary table for finding what a project owns.

## Column Relationships

- **(artifact_id, project_id) is the natural key** — an artifact can belong to multiple projects (rare but possible across registries), and a project typically owns many artifacts.
- **`artifact_source` identifies the type of artifact** — `GITHUB` for repos, `NPM` for packages, chain names (`OPTIMISM`, `BASE`, etc.) for contracts/addresses.
- **`project_source` identifies the registry** — same artifact may be claimed by different projects in different registries.

## Query Rules

### Filtering Rules

- **Filter by `artifact_source = 'GITHUB'`** to get only GitHub repositories. This is the most common filter for repo-focused analysis.
- **Filter by `project_source = 'OSS_DIRECTORY'`** when working with OSSD data. Without this, you will get artifacts claimed by OP Atlas, DeFiLlama, and other registries, leading to duplicates.
- **Combine both filters** for the cleanest OSSD repo analysis: `WHERE project_source = 'OSS_DIRECTORY' AND artifact_source = 'GITHUB'`.
- **Bucket partitioning on `artifact_id`** — filtering on `artifact_id` can leverage partition pruning for better performance.

### Aggregation Rules

- **`COUNT(DISTINCT artifact_id) WHERE artifact_source = 'GITHUB'`** to count repos per project.
- **`COUNT(DISTINCT artifact_id) WHERE artifact_source IN ('OPTIMISM', 'BASE', ...)`** to count deployed contracts.
- **Always include source filters** in aggregations to avoid double-counting across registries.

### Join Rules

- **Join to `oso.repositories_v0` on `artifact_id`** to get enriched repo metadata (stars, forks, language, license).
- **Join to `oso.sboms_v0` on `artifact_id = dependent_artifact_id`** to find what packages a project's repos depend on.
- **Join to `oso.projects_v1` on `project_id`** for project display names.
- **Join to `oso.projects_by_collection_v1` on `project_id`** to filter by ecosystem.
- **Join to event/metric tables on `artifact_id` or `project_id`** for activity data.
- **Cardinality**: many-to-many (one project has many artifacts; an artifact can appear under multiple project sources).

### Common Patterns

**List all GitHub repos for an OSSD project:**
```sql
SELECT
  artifact_id,
  artifact_namespace,
  artifact_name
FROM oso.artifacts_by_project_v1
WHERE project_source = 'OSS_DIRECTORY'
  AND project_name = 'uniswap'
  AND artifact_source = 'GITHUB'
```

**Count repos per OSSD project:**
```sql
SELECT
  project_name,
  COUNT(DISTINCT artifact_id) AS repo_count
FROM oso.artifacts_by_project_v1
WHERE project_source = 'OSS_DIRECTORY'
  AND artifact_source = 'GITHUB'
GROUP BY project_name
ORDER BY repo_count DESC
LIMIT 50
```

**Find which OSSD projects have deployed on a specific chain:**
```sql
SELECT DISTINCT
  project_name
FROM oso.artifacts_by_project_v1
WHERE project_source = 'OSS_DIRECTORY'
  AND artifact_source = 'OPTIMISM'
```

**Get all artifact types for a project:**
```sql
SELECT
  artifact_source,
  COUNT(DISTINCT artifact_id) AS artifact_count
FROM oso.artifacts_by_project_v1
WHERE project_source = 'OSS_DIRECTORY'
  AND project_name = 'uniswap'
GROUP BY artifact_source
ORDER BY artifact_count DESC
```

## Business Logic

This mart projects from `int_artifacts_by_project`, which is the result of a complex multi-source union and discovery pipeline:

1. **Direct registrations** from OSSD, OP Atlas, OpenDevData, DeFiLlama, and Open Labels Initiative — each source contributes its own artifact-to-project mappings.
2. **Downstream discovery** — OSSD and OP Atlas artifacts are extended with blockchain contract deployments traced from registered deployer addresses.
3. **Package discovery** — repos registered to a project are matched to NPM/PyPI packages they maintain via `int_packages__current_maintainer_only`.

All sources are unioned and deduplicated within `int_artifacts_by_project_all_sources`, then joined to `int_projects` for project metadata.

## Best Practices

- **Use this table instead of `oso.artifacts_v1`** — `artifacts_v1` is a massive table of all known artifacts (including those not attributed to any project) and is much larger. `artifacts_by_project_v1` gives you only artifacts that are owned by a known project, which is almost always what you want.
- **Always filter by `project_source = 'OSS_DIRECTORY'` and `artifact_source = 'GITHUB'`** for OSSD repo analysis. These two filters together give you the cleanest view of OSSD project repositories.
- **Use `artifact_id` as the join key** to downstream tables like `repositories_v0` and `sboms_v0` — it is the canonical artifact identifier across all OSO tables.
- **This is the bridge table** between the project hierarchy (projects, collections) and the artifact/repo/dependency hierarchy (repositories, SBOMs, packages). Most analysis pipelines start here.
