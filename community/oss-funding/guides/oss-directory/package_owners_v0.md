## Column Relationships

- **(package_artifact_id, package_owner_artifact_id) is the natural key** — a package can have multiple owner repos (rare), and a repo can own multiple packages.
- **`package_artifact_id` maps to `package_artifact_id` in `sboms_v0`** — this is the link between "what depends on this package" and "who maintains it".
- **`package_owner_artifact_id` maps to `artifact_id` in `repositories_v0` and `artifacts_by_project_v1`** — this links package ownership back to the project hierarchy.

## Query Rules

### Filtering Rules

- **Filter by `package_artifact_source`** to scope to a specific package ecosystem.
- **Filter by `package_owner_artifact_namespace`** to find all packages maintained by a specific GitHub org.

### Aggregation Rules

- **`COUNT(DISTINCT package_artifact_id)` per owner repo** for package count per maintainer.
- **`COUNT(DISTINCT package_owner_artifact_id)` per package** to identify packages with multiple maintainers.

### Join Rules

- **Join to `oso.sboms_v0` on `package_artifact_id`** to trace dependencies through to their maintainers: repo → depends on package → maintained by owner repo.
- **Join to `oso.artifacts_by_project_v1` on `package_owner_artifact_id = artifact_id`** to attribute package ownership to OSSD projects.
- **Join to `oso.repositories_v0` on `package_owner_artifact_id = artifact_id`** to get metadata about the maintainer repo.
- **Cardinality**: typically one owner per package, but some packages have multiple maintainer repos. Expect low fan-out.

### Common Patterns

**Find all packages maintained by repos of an OSSD project:**
```sql
SELECT
  po.package_artifact_source,
  po.package_artifact_namespace,
  po.package_artifact_name
FROM oso.package_owners_v0 AS po
INNER JOIN oso.artifacts_by_project_v1 AS a
  ON po.package_owner_artifact_id = a.artifact_id
WHERE a.project_source = 'OSS_DIRECTORY'
  AND a.project_name = 'openzeppelin'
  AND a.artifact_source = 'GITHUB'
ORDER BY po.package_artifact_source, po.package_artifact_name
```

**Trace the full dependency chain: project A depends on packages maintained by project B:**
```sql
SELECT DISTINCT
  a_dependent.project_name AS dependent_project,
  s.package_artifact_name AS package_name,
  a_owner.project_name AS owner_project
FROM oso.sboms_v0 AS s
INNER JOIN oso.artifacts_by_project_v1 AS a_dependent
  ON s.dependent_artifact_id = a_dependent.artifact_id
INNER JOIN oso.package_owners_v0 AS po
  ON s.package_artifact_id = po.package_artifact_id
INNER JOIN oso.artifacts_by_project_v1 AS a_owner
  ON po.package_owner_artifact_id = a_owner.artifact_id
WHERE a_dependent.project_source = 'OSS_DIRECTORY'
  AND a_dependent.project_name = 'optimism'
  AND a_dependent.artifact_source = 'GITHUB'
  AND a_owner.project_source = 'OSS_DIRECTORY'
  AND a_owner.artifact_source = 'GITHUB'
ORDER BY owner_project, package_name
```

**Most-depended-on OSSD projects (by package usage):**
```sql
SELECT
  a_owner.project_name AS package_owner_project,
  COUNT(DISTINCT s.dependent_artifact_id) AS dependent_repo_count,
  COUNT(DISTINCT po.package_artifact_id) AS package_count
FROM oso.package_owners_v0 AS po
INNER JOIN oso.sboms_v0 AS s
  ON po.package_artifact_id = s.package_artifact_id
INNER JOIN oso.artifacts_by_project_v1 AS a_owner
  ON po.package_owner_artifact_id = a_owner.artifact_id
WHERE a_owner.project_source = 'OSS_DIRECTORY'
  AND a_owner.artifact_source = 'GITHUB'
GROUP BY a_owner.project_name
ORDER BY dependent_repo_count DESC
LIMIT 30
```

## Business Logic

This mart is a `SELECT DISTINCT` over `int_sbom_to_packages`, the same upstream table that feeds `sboms_v0`. While `sboms_v0` focuses on the dependency side (repo → package), `package_owners_v0` focuses on the ownership side (package → maintainer repo).

The ownership mapping comes from deps.dev (`int_packages_from_deps_dev` → `int_packages__current_maintainer_only`), which resolves package registry entries to their source GitHub repositories. Only the current maintainer is retained — historical ownership changes are not tracked.

## Lineage

### Full Chain
```
deps.dev package data → oso.stg_deps_dev__packages
                          → oso.int_packages_from_deps_dev
                            → oso.int_packages__current_maintainer_only ─┐
OSSD SBOM data → oso.int_sbom__latest_snapshot ─────────────────────────┼→ oso.int_sbom_to_packages
oso.seed_package_source_mapping ────────────────────────────────────────┘    → oso.package_owners_v0
```

### Upstream (direct)
- `oso.int_sbom_to_packages` — resolves SBOM entries to package-owner mappings

### Downstream (known consumers)
- Supply chain analysis (who maintains what I depend on)
- Package influence scoring
- Cross-project dependency graphs

## Best Practices

- **Combine with `sboms_v0` for full dependency chain analysis** — `sboms_v0` tells you what a repo depends on, `package_owners_v0` tells you who maintains those dependencies. Together they answer "which OSSD projects does my project depend on?"
- **Always join to `artifacts_by_project_v1`** when attributing package ownership to projects — `package_owners_v0` only identifies the maintainer repo, not the OSSD project.
- **Ownership is current-maintainer-only** — if a package was transferred between repos, only the current maintainer appears. Historical ownership is not tracked.
- **Coverage depends on deps.dev** — packages not indexed by deps.dev will not appear in this table.
