# OSS Directory Data Guide

> Hub guide for all models derived from the OSS Directory (OSSD) — the curated registry of open source projects, their repositories, and software dependencies tracked by OSO.

## What is OSS Directory?

[OSS Directory](https://github.com/opensource-observer/oss-directory) is a curated registry of open source projects maintained as YAML files in a GitHub repo. It defines:

- **Projects** — individual open source projects (eg Uniswap, Aave, Optimism)
- **Collections** — groups of related projects (eg ecosystems, funding rounds)
- **Artifacts** — the repos, packages, contracts, and addresses owned by each project

OSS Directory is one of several registries in OSO (alongside OP Atlas, DeFiLlama, OpenDevData, and Open Labels Initiative). When querying OSSD-sourced data, always filter by:

| Table | Filter |
|-------|--------|
| `projects_v1` | `project_source = 'OSS_DIRECTORY'` |
| `collections_v1` | `collection_source = 'OSS_DIRECTORY'` |
| `projects_by_collection_v1` | `collection_source = 'OSS_DIRECTORY'` |
| `artifacts_by_project_v1` | `project_source = 'OSS_DIRECTORY'` (and `artifact_source = 'GITHUB'` for repos) |

Without these filters, you will get data from other registries mixed in, leading to duplicates and inflated counts.

## Data Pipeline

The OSSD pipeline runs in Dagster and follows this sequence:

```
oss-directory repo (YAML files)
  │
  ├─→ ossd.projects          ─→ stg_ossd__current_projects     ─→ int_projects
  ├─→ ossd.collections       ─→ stg_ossd__current_collections  ─→ int_collections
  │                                                              ─→ int_projects_by_collection
  │
  ├─→ ossd.repositories      ─→ stg_ossd__current_repositories ─→ int_repositories__ossd
  │   (GitHub API enrichment:
  │    stars, forks, language,
  │    license, fork status)
  │
  └─→ ossd.sbom              ─→ stg_ossd__current_sbom         ─→ int_sbom_from_ossd
      (GitHub SBOM API:                                          ─→ int_sbom__latest_snapshot
       dependency manifests)                                     ─→ int_sbom_to_packages
                                                                    (joined with deps.dev
                                                                     package metadata)
```

**Key details:**
- **Projects & collections** are fetched directly from the oss-directory YAML files. The pipeline skips writes if the repo SHA hasn't changed since last materialization.
- **Repositories** are enriched via the GitHub REST API — the pipeline extracts GitHub URLs from project YAML, then fetches metadata (stars, forks, language, license, etc.) for each repo.
- **SBOMs** are fetched via the GitHub SBOM API for repositories that haven't been refreshed in 7 days (configurable). Dependency manifests (package.json, requirements.txt, Cargo.toml, go.mod, etc.) are parsed into package references.
- **Package resolution** happens in the sqlmesh layer: SBOM entries are matched to known packages from [deps.dev](https://deps.dev) using `seed_package_source_mapping`, which bridges package manager naming conventions to OSO artifact identifiers.

## Entity Hierarchy

```
collections_v1
  └─ projects_by_collection_v1 (many-to-many)
       └─ projects_v1
            └─ artifacts_by_project_v1 (one-to-many)
                 └─ repositories_v0 (enriched GitHub metadata)
                      └─ sboms_v0 (repo → depends on package)
                           └─ package_owners_v0 (package → maintained by repo)
```

## Model Reference

| Model | Guide | Grain | Description |
|-------|-------|-------|-------------|
| `oso.collections_v1` | [collections_v1.md](collections_v1.md) | collection_id | Collection metadata (ecosystems, funding rounds) |
| `oso.projects_v1` | [projects_v1.md](projects_v1.md) | project_id | Project metadata (name, description) |
| `oso.projects_by_collection_v1` | [projects_by_collection_v1.md](projects_by_collection_v1.md) | (project_id, collection_id) | Which projects belong to which collections |
| `oso.artifacts_by_project_v1` | [artifacts_by_project_v1.md](artifacts_by_project_v1.md) | (artifact_id, project_id) | Which artifacts (repos, packages, contracts) belong to which projects |
| `oso.repositories_v0` | [repositories_v0.md](repositories_v0.md) | artifact_id | Enriched GitHub repo metadata (stars, forks, language, license) |
| `oso.sboms_v0` | [sboms_v0.md](sboms_v0.md) | (dependent_artifact_id, package_artifact_id) | Repo-to-package dependency edges |
| `oso.package_owners_v0` | [package_owners_v0.md](package_owners_v0.md) | (package_artifact_id, package_owner_artifact_id) | Package-to-maintainer-repo mapping |

## ID Generation

All entity IDs in OSO are deterministic hashes generated by `@oso_entity_id(source, namespace, name)`. This means:

- The same entity always gets the same ID
- Different registries produce different IDs for the same real-world project (eg OSSD's "uniswap" and OP Atlas's "uniswap" have different `project_id` values)
- `*_id` fields are OSO-internal — they are NOT the same as `*_source_id` (the native ID from GitHub, NPM, etc.)
- Always use `*_id` for joins across OSO tables

## Common Query Patterns

### Full OSSD hierarchy: collection → projects → repos

```sql
SELECT
  c.collection_name,
  p.project_name,
  r.artifact_namespace AS github_org,
  r.artifact_name AS repo_name,
  r.star_count,
  r.language
FROM oso.projects_by_collection_v1 AS c
INNER JOIN oso.artifacts_by_project_v1 AS a
  ON c.project_id = a.project_id
INNER JOIN oso.repositories_v0 AS r
  ON a.artifact_id = r.artifact_id
WHERE c.collection_source = 'OSS_DIRECTORY'
  AND c.collection_name = 'ethereum'
  AND a.project_source = 'OSS_DIRECTORY'
  AND a.artifact_source = 'GITHUB'
  AND r.is_fork = false
ORDER BY r.star_count DESC
```

### Supply chain: which OSSD projects does my project depend on?

```sql
SELECT DISTINCT
  a_owner.project_name AS dependency_project,
  po.package_artifact_source AS package_ecosystem,
  po.package_artifact_name AS package_name
FROM oso.artifacts_by_project_v1 AS a_dep
INNER JOIN oso.sboms_v0 AS s
  ON a_dep.artifact_id = s.dependent_artifact_id
INNER JOIN oso.package_owners_v0 AS po
  ON s.package_artifact_id = po.package_artifact_id
INNER JOIN oso.artifacts_by_project_v1 AS a_owner
  ON po.package_owner_artifact_id = a_owner.artifact_id
WHERE a_dep.project_source = 'OSS_DIRECTORY'
  AND a_dep.project_name = 'optimism'
  AND a_dep.artifact_source = 'GITHUB'
  AND a_owner.project_source = 'OSS_DIRECTORY'
  AND a_owner.artifact_source = 'GITHUB'
ORDER BY dependency_project, package_name
```

### Most-depended-on packages in an ecosystem

```sql
SELECT
  s.package_artifact_source AS ecosystem,
  s.package_artifact_name AS package_name,
  COUNT(DISTINCT a.project_id) AS dependent_project_count,
  COUNT(DISTINCT s.dependent_artifact_id) AS dependent_repo_count
FROM oso.sboms_v0 AS s
INNER JOIN oso.artifacts_by_project_v1 AS a
  ON s.dependent_artifact_id = a.artifact_id
INNER JOIN oso.projects_by_collection_v1 AS c
  ON a.project_id = c.project_id
WHERE c.collection_source = 'OSS_DIRECTORY'
  AND c.collection_name = 'ethereum'
  AND a.project_source = 'OSS_DIRECTORY'
  AND a.artifact_source = 'GITHUB'
GROUP BY s.package_artifact_source, s.package_artifact_name
ORDER BY dependent_project_count DESC
LIMIT 30
```

## Gotchas

- **Always filter by source** — without `project_source = 'OSS_DIRECTORY'` (or equivalent), you get data from multiple registries mixed together. This is the #1 mistake.
- **Use `artifacts_by_project_v1`, not `artifacts_v1`** — `artifacts_v1` contains ALL known artifacts (including those not attributed to any project) and is much larger. `artifacts_by_project_v1` gives you only project-attributed artifacts.
- **SBOM coverage is OSSD-only** — only repos registered in OSS Directory have their SBOMs parsed. Repos from other registries are not included.
- **Package resolution depends on deps.dev** — packages not indexed by deps.dev won't appear in `sboms_v0` or `package_owners_v0`.
- **Repository metadata is point-in-time** — star/fork counts reflect the last pipeline refresh (repos are re-fetched every ~18 hours), not real-time GitHub values.
- **SBOMs refresh weekly** — repos are re-scanned every 7 days by default. Recently added dependencies may not appear immediately.
