## Column Relationships

- **`collection_id` is deterministic** — generated by `@oso_entity_id(collection_source, collection_namespace, collection_name)`. All OSO entity IDs follow this pattern: hash the (source, namespace, name) tuple.
- **`collection_source` controls which registry the collection comes from** — multiple sources can define collections independently.
- **`collection_name` is unique within a source** but not globally unique.

## Query Rules

### Filtering Rules

- **Filter by `collection_source = 'OSS_DIRECTORY'`** to scope to OSSD-curated collections. Without this filter, you will get collections from OP Atlas, DeFiLlama, and other sources.
- **Use `display_name` for keyword searches** — it contains the human-readable name (eg `Retro Funding 4`), while `collection_name` is a slug (eg `retro-funding-4`).

### Aggregation Rules

- **`COUNT(*)` per `collection_source`** to see how many collections each registry contributes.
- **Always include `collection_source` in GROUP BY** to avoid mixing registries.

### Join Rules

- **Join to `oso.projects_by_collection_v1` on `collection_id`** to find all projects in a collection.
- **Cardinality**: one row per collection. Joining to `projects_by_collection_v1` fans out to one row per project membership.

### Common Patterns

**List all OSSD collections:**
```sql
SELECT
  collection_id,
  collection_name,
  display_name,
  description
FROM oso.collections_v1
WHERE collection_source = 'OSS_DIRECTORY'
ORDER BY display_name
```

**Search collections by keyword:**
```sql
SELECT
  collection_id,
  collection_source,
  collection_name,
  display_name
FROM oso.collections_v1
WHERE collection_source = 'OSS_DIRECTORY'
  AND LOWER(display_name) LIKE '%ethereum%'
```

**Count collections per source:**
```sql
SELECT
  collection_source,
  COUNT(*) AS collection_count
FROM oso.collections_v1
GROUP BY collection_source
ORDER BY collection_count DESC
```

## Business Logic

This is a thin projection over `int_collections`, which aggregates collection records from multiple registries:

- **OSS Directory** (`stg_ossd__current_collections`) — curated YAML files defining ecosystems and categories
- **OP Atlas** (`int_projects_by_collection_in_op_atlas`) — Optimism's funding round groupings
- **DeFiLlama** (`int_projects_by_collection_in_defillama`) — DeFi protocol categories
- **Open Labels Initiative** (`int_projects_by_collection_in_openlabelsinitiative`) — community-sourced groupings

Collections from different sources are not deduplicated — an "Ethereum" collection may exist in both OSS Directory and Open Labels Initiative as separate rows.

## Lineage

### Upstream (direct)
- `oso.int_collections` — unions collection records from all source registries

### Downstream (known consumers)
- `oso.projects_by_collection_v1` — joins collections to their member projects

## Best Practices

- **Always filter by `collection_source = 'OSS_DIRECTORY'`** when working with OSSD ecosystem data to avoid mixing registries.
- **Use `collection_id` for joins**, not `collection_name` — names are only unique within a source.
- **This table is metadata only** — join to `projects_by_collection_v1` to see which projects belong to a collection.
