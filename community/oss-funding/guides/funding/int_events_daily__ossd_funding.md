## Event Source Mapping

The `event_source` is derived from the raw `from_funder_name` and `grant_pool_name` fields:

| from_funder_name | grant_pool_name pattern | event_source |
|------------------|------------------------|--------------|
| `optimism` | `%retro%` | `OPTIMISM_RETROFUNDING` |
| `optimism` | `%season%` | `OPTIMISM_GOVGRANTS` |
| `%octant%` | any | `OCTANT` |
| `%gitcoin%` | any | `GITCOIN` |
| `%dao-drops%` | any | `DAODROPS` |
| anything else | any | `UPPER(from_funder_name)` (eg `STELLAR`, `ARBITRUMFOUNDATION`, `CLRFUND`) |

## Column Relationships

- **`to_artifact_id` = `project_id` in `projects_v1`** when `project_source = 'OSS_DIRECTORY'` — the ID is generated from the OSSD project slug.
- **`from_artifact_id`** is the funder entity in the OSSD namespace — there are roughly 7 distinct funder IDs.
- **`event_source` is derived, not stored in the CSV** — it is a CASE expression over `from_funder_name` and `grant_pool_name`.
- **Amounts are summed per (day, source, type, from, to)** — the GROUP BY aggregation collapses multiple CSV rows on the same day for the same project/funder.

## Query Rules

### Filtering Rules

- **Filter by `event_source`** to scope to a specific funder (eg `WHERE event_source = 'OCTANT'`).
- **No need to filter `event_type`** — it is always `FUNDING_AWARDED` in this table.

### Join Rules

- **Join to `oso.projects_v1` on `to_artifact_id = project_id`** with `WHERE project_source = 'OSS_DIRECTORY'`.
- **Join to `oso.projects_by_collection_v1`** to group by ecosystem.
- Same join patterns as `int_events_daily__funding` — see hub guide.

### Common Patterns

**Octant funding by project:**
```sql
SELECT
  p.project_name,
  p.display_name,
  SUM(f.amount) AS octant_funding_usd
FROM oso.int_events_daily__ossd_funding AS f
INNER JOIN oso.projects_v1 AS p
  ON f.to_artifact_id = p.project_id
WHERE p.project_source = 'OSS_DIRECTORY'
  AND f.event_source = 'OCTANT'
GROUP BY p.project_name, p.display_name
ORDER BY octant_funding_usd DESC
```

**Funding by funder and year:**
```sql
SELECT
  event_source,
  YEAR(bucket_day) AS year,
  SUM(amount) AS total_usd,
  COUNT(DISTINCT to_artifact_id) AS projects_funded
FROM oso.int_events_daily__ossd_funding
GROUP BY event_source, YEAR(bucket_day)
ORDER BY event_source, year
```

**Projects funded by multiple OSSD funders:**
```sql
SELECT
  p.project_name,
  COUNT(DISTINCT f.event_source) AS num_funders,
  ARRAY_AGG(DISTINCT f.event_source) AS funders,
  SUM(f.amount) AS total_usd
FROM oso.int_events_daily__ossd_funding AS f
INNER JOIN oso.projects_v1 AS p
  ON f.to_artifact_id = p.project_id
WHERE p.project_source = 'OSS_DIRECTORY'
GROUP BY p.project_name
HAVING COUNT(DISTINCT f.event_source) > 1
ORDER BY num_funders DESC, total_usd DESC
```

## Business Logic

This model transforms the oss-funding CSV data from `stg_ossd__current_funding` into the daily event format used by the unified funding table.

The staging table reads from `bigquery.ossd.funding`, which is populated by the Dagster OSSD pipeline. It ingests the consolidated `data/funding_data.csv` from the [oss-funding repo](https://github.com/opensource-observer/oss-funding) — a curated registry maintained per-funder with YAML profiles and CSV uploads.

The transformation:
1. **Parses `funding_date`** — handles multiple date formats, defaults to 1970-01-01 on failure.
2. **Maps `from_funder_name` to `event_source`** — uses CASE logic on funder name + grant pool name (see mapping above).
3. **Generates entity IDs** — `@oso_entity_id('OSS_DIRECTORY', 'oso', ...)` for both funder and project.
4. **Aggregates to daily grain** — GROUP BY (day, source, type, from, to), SUM(amount).

## Best Practices

- **Use the unified `int_events_daily__funding` table** for cross-funder analysis — query this table directly only when you need OSSD-specific data or want to avoid Open Collective/Gitcoin data.
- **`event_source` mapping is pattern-based** — if new funders are added to oss-funding with non-standard names, they will fall through to `UPPER(from_funder_name)`.
- **Funding dates depend on CSV quality** — some records may have imprecise dates. The staging layer defaults to 1970-01-01 for unparseable dates.
- **Amounts are USD at award time** — no token price adjustments.
